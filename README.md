# eml-sr_study: EML演算子を用いたシンボリック回帰の検証

本プロジェクトは、単一の二項演算子「EML演算子」を用いたシンボリック回帰アルゴリズムの性能検証と、既存ライブラリの比較評価を目的としています。

## EML演算子とシンボリック回帰について

今年(26年)の3月、単一の二項演算子のみを用いてあらゆる初等関数を表現できる手法が提案されました（arXiv:2603.21852に基づく）。この中核となる「EML演算子」は以下のように定義されます。

$$ \text{eml}(x, y) = \exp(x) - \ln(y) $$

この演算子の画期的な点は、四則演算（ $+, -, \times, \div$ ）や三角関数（ $\sin, \cos$ ）、その他の基本的な初等関数を、この `eml(x, y)` をノードとする単一の均一な二分木構造のネストのみで近似的に表現できるという数学的性質にあります。

**EML演算子を用いたシンボリック回帰の利点**
従来のシンボリック回帰（遺伝的プログラミング等を用いる手法）では、探索空間に多様な演算子（ $+, \times, \sin$ など）が混在するため、探索空間が離散的かつ複雑になり、最適化が困難でした。
一方、EML演算子を用いた手法では、数式を「すべて同じノード（eml）で構成された均一な二分木」として表現できます。これにより、構造全体が微分可能となり、Adamなどの勾配ベースの連続的な最適化手法（ディープラーニングで用いられる手法）を直接適用できるようになるという大きな利点があります。つまり、数式の構造探索を連続的な最適化問題として扱い、未知の数式をより効率的かつ滑らかに探索・発見できる可能性を秘めています。

## 第一部：EML演算子関連ライブラリの比較と選定

EML演算子の概念を用いたシンボリック回帰の実装（クレート・ライブラリ）は、GitHub上にいくつか公開されています。本研究では、代表的な5つのリポジトリ（gh1〜gh5）を調査し、それぞれの特徴や実装アプローチを比較した上で、実験に用いるライブラリを選定しました。

### 各リポジトリの特徴

1. **gh1 (arlegotin/eml-symbolic-regression)**
   - 論文の概念実証（PoC）に近い初期の実装。主にPythonで記述されており、EML演算子の基本的な挙動や勾配計算の仕組みを理解するには適しています。しかし、深い木構造を扱う際の計算速度やメモリ効率の面でボトルネックがあり、大規模なデータセットへの適用には実用上の課題があります。
2. **gh2 (oaustegard/eml-sr(pytorch))**
   - PyTorch等の既存の深層学習フレームワークを活用し、自動微分を活かした実装。しかし、パッケージとしてインストールして手軽に試すことが難しく、リポジトリを直接クローンして環境構築を行う必要があったため、今回の実験における比較検討の対象からは除外しました。
3. **gh3 (Amin7410/Project-Andrzej,eml-sr(rust))**
   - **Rust言語で実装されたクレート。** 高速な実行速度と強固なメモリ安全性を備えつつ、Pythonからのバインディングも提供されているため、実験スクリプト（データ前処理等）との統合が非常に容易です。探索木の構築や勾配評価がシステムレベルで高度に最適化されており、実用的なシンボリック回帰ライブラリとして現時点で最も完成度が高いです。
4. **gh4 (cool-japan/oxieml)**
   - 独自の最適化ヒューリスティクスや進化計算を組み合わせたハイブリッド探索を試みているリポジトリ。アプローチは野心的ですが、まだ開発の初期段階にあり、安定した動作やドキュメントの充実度という点で他のプロジェクトに見劣りします。
5. **gh5 (janegbert/pyeml)**
   - 他の機械学習手法（ニューラルネットワークの層など）とEMLを組み合わせた複雑なアーキテクチャ。高い表現力を持ちますが、モデルの解釈可能性（最終的に綺麗な数式を抽出できるか）がやや難しく、純粋なシンボリック回帰ツールというよりは表現学習モデルに近い位置づけとなっています。

### 選定理由と考察
上記の実装を比較検討した結果、本実験では **gh3 (eml-sr)** を採用することに決定しました。
最大の理由は、Rustによる実装がもたらす「圧倒的な計算パフォーマンス」です。EML演算子による探索は、複雑な関数を表現するために深い木構造を必要とする場合があり、計算コストが指数関数的に増大する傾向があります。gh3は、この計算ボトルネックをRustの実行速度によってカバーしています。さらに、Pythonエコシステムからシームレスに呼び出せる利便性も備えており、Feynman Equationsのような多様なデータセットに対して、現実的な時間内で網羅的な実験・評価を行うための基盤として最も優れていると判断しました。

## 第二部：eml-sr (gh3) によるFeynman Equationsの推定実験

選定した `eml-sr` を用い、物理学の基本的な方程式群である「Feynman Equations」の99個の数式に対して、データのみから数式構造を自動発見できるかを検証しました。

**検証手法（Methodology）**
分析時は真の数式（正解）をアルゴリズムに対して「未知」として扱い、各方程式の定義された変数範囲からランダムにサンプリング(今回は誤差なし)したデータポイント（変数 $x_i$ と目的変数 $y$ のペア）のみを `eml-sr` に入力しました。純粋なデータ駆動型アプローチで、与えられた入出力データからどこまで真の物理法則の構造に迫れるかをテストしています。

実験は、標準的な探索パラメータ（前半）と、探索幅を大幅に広げた高精度パラメータ（後半）の二段階で実施し、レポート（`texts/claude_allfunc_estimate_sr_report.md`, `texts/allfunc_moreestimate_sr_report.md`）にまとめました。

### 前半：標準パラメータによるベースライン検証
最初のフェーズでは、標準的なハイパーパラメータ（サンプル数: 200, ビーム幅: 300, 最大複雑度: 6）を使用し、全99式に対して回帰を実行しました（実行時間：約49分）。

- **結果と考察**:
  - 99式中 **9式 (9.1%)** で完全な数式の回収（RMSE < 1e-4）に成功しました。
  - 成功した式には、単純な積や商（例: $\mu \cdot N$, $q/C$）だけでなく、スネルの法則（ $\arcsin(n \cdot \sin(\theta_2))$ ）やブラッグの法則など、`arcsin` を含む複雑な三角関数の合成式も含まれていました。EML演算子が特定の非線形パターンを非常に少ない複雑度（ノード数）で見事に捉えられることが確認できました。
- **直面した課題**:
  - 4変数以上の多変数方程式や、正規分布型（Gaussian）のように指数内に二乗を含む構造、ローレンツ因子（ $\sqrt{1 - v^2/c^2}$ ）のような相対論的構造は、最大複雑度6では表現しきれず失敗しました。
  - また、 $1/2$ や $3/2$ といった有理数の係数を含む式も、デフォルトの定数のみでEMLノードを組み合わせて表現しようとするため失敗するケースが目立ちました。

### 後半：高精度パラメータによる限界検証（ビーム幅とサンプル数の拡張）
前半の課題を受け、第二フェーズでは木構造の最大複雑度は「6」のまま据え置き、探索の多様性を担保するビーム幅を 1000、サンプル数を 750 へと大幅に引き上げて再実験を行いました（実行時間：約126分）。

- **結果と考察**:
  - 計算リソースと時間を約2.5倍投入したにも関わらず、完全に回収できた数式は**前半と全く同じ9式**にとどまりました。
  - この結果から、探索幅（ビーム幅）を広げても発見できる数式が増えないことが明らかになりました。つまり、最適化が局所解に陥っているのではなく、**「最大複雑度（木の深さ）の不足」と「定数表現に複雑度を浪費してしまうこと」が、表現力の絶対的なボトルネックになっている**ことがわかりました。

## 考察と今後の課題

本プロジェクトの検証を通じて、EML演算子を用いたシンボリック回帰が持つ「構造の均一化による連続的最適化」という理論的な強みと、`arcsin` のような複雑な構造を低複雑度で見事に発見できる高いポテンシャルを確認できました。一方で、ビーム幅を広げても成功数が増加しなかったという実験結果から、最適化手法の弱さよりも **「表現力の限界（複雑度制限と定数の浪費）」** が最大のボトルネックであることが判明しました。

現在の `eml-sr` は、数式内の係数や定数をEML演算子の組み合わせによって近似的に表現しようとするため、本来の変数間の関係性を構築するための「複雑度（木の深さ）」を定数のために大量に消費してしまっています。これが原因で、複雑な数式が表現できなくなっています。

今後はこの課題を解決するため、定数に余計な複雑度を消費しないよう、定数をすべて小数（スカラー値）として直接表現・最適化できるように、`eml-sr` のソースコード（Rust実装）を直接編集・改修していくことがプロジェクトの次のステップとなります。
